## Pour un agent qui AGIT sur de l'argent réel : le LLM propose une intention JSON, du code testé décide — pas de tool-calling — 2026-07-17
- **Choix d'archi (concierge I-10)** : le LLM ne décide RIEN. Il émet `{action, amountEur, ...}` en `json_object`, et une fonction pure `decideAction()` (23 tests dont adversariaux) valide/tranche. La validation est du **code qu'on contrôle**, pas une promesse du modèle. Défaut = proposer, pas exécuter (une seule action live : code promo, zéro argent sortant).
- **Pourquoi PAS de tool-calling** : le projet n'en utilise nulle part (`grep tool_calls|function_call` = 0), et `_llm.js` construit un body IDENTIQUE pour 7 providers. Ajouter `tools` casserait la cascade (Cloudflare Workers AI ne suit pas le format OpenAI) pour un gain nul ici — `json_object` + dispatcher validé côté serveur suffit et marche même avec `llama-3.1-8b-instant`. Le seul mécanisme de sortie structurée du projet reste `responseFormat:{type:"json_object"}` + parsing manuel avec regex de secours (`{[\s\S]*}`).
- **Garde-fous réutilisables** (mêmes conventions que `social-webhook`/`guide-write`) : `CONCIERGE_MODE=shadow` par défaut / `live` explicite · kill-switch `CONCIERGE_DISABLED` (⚠️ accepter `"1"` ET `"true"` — `social-webhook` ne gère que `"1"`, piège connu) · plafond borné · **un dépassement ESCALADE, ne s'écrête jamais en silence**. C'est le 4e système du projet à adopter shadow-par-défaut (après RM advisory, agents-execute, social bot) — c'est la convention maison pour tout ce qui touche à du réel.

## Statut `agent_actions` pour rejeter une reco = `bloqué`, PAS `ignoré` (validateur PATCH resserré, memory périmée corrigée) — 2026-07-15
- **Piège** : une memory de feedback prescrivait `status='ignoré'` pour toute reco à ne pas exécuter. En l'appliquant, le PATCH `/api/agents-actions` a rejeté la requête — le validateur actuel n'accepte que `["backlog","en-cours","fait","bloqué","a-planifier"]`. Les 25 lignes historiques `status='ignoré'` visibles en D1 (`stats` globales) viennent d'une version antérieure du endpoint, jamais nettoyée — leur présence en base a fait croire, à tort, que ce statut était toujours valide.
- **La prochaine fois** : `agents-triage.js` (cron hebdo automatique) utilise déjà `bloqué` pour ses propres détections — aligner tout tri manuel sur ce même vocabulaire. Plus généralement : une memory qui cite un statut/paramètre exact doit être re-vérifiée contre le validateur RÉEL du endpoint avant d'agir, pas juste contre des données déjà en base (qui peuvent dater d'avant un resserrement de règles).

## 🔴 `files_hint` (agents-run.js) n'est qu'une liste de CHEMINS en texte — jamais le contenu du fichier — 2026-07-04
- **Piège trouvé en investiguant un doublon** : voyageur-research a régénéré à l'identique 4 personas déjà écrits dans `docs/marketing/atelier-acquisition-roi-2026-06.md`. Cause racine : `files_hint` de chaque agent (ex. `"D1 google_reviews / contacts, functions/api/google-reviews.js, ~/.claude/skills/voyageur-research-amaryllis/SKILL.md"`) est injecté tel quel dans le prompt (`FICHIERS CLÉS à analyser : ${agent.files_hint}`) — c'est du texte informatif pour un humain qui relit la config, PAS un contenu réellement lu par le LLM. Aucun agent n'a jamais eu accès au contenu réel de `docs/*.md` (117 fichiers) : ni via `files_hint` (chemins seuls), ni via le RAG (qui n'ingérait que faits biens + avis + captions avant ce jour).
- **La prochaine fois** : ne jamais supposer qu'un agent "connaît" un doc parce qu'il est mentionné dans son `files_hint` ou qu'il existe dans le repo — seul ce qui est explicitement injecté dans le prompt (RAG, PLAYBOOK_DIGEST, FISCAL_CONTEXT, historique D1) est réellement visible. Avant de reprocher à un agent un doublon/une hallucination factuelle, vérifier D'ABORD s'il avait ne serait-ce que la POSSIBILITÉ d'accéder à l'info manquante.

## 🔴 Cloudflare Vectorize : id de vecteur limité à 64 OCTETS (pas de long chemin/slug) — 2026-07-04
- **Piège trouvé en ingérant réellement en prod** (pas en dry-run) : des ids construits en slugifiant un chemin de fichier (`doc-docs-marketing-emails-prospection-institutionnels-2026-06-md-0`) dépassaient 64 octets → `VECTOR_UPSERT_ERROR 40008`, faisant échouer TOUT le batch de 50 (dont des items valides). Fix : hash court (SHA-1 tronqué à 12 hex) + index, longueur bornée quel que soit le chemin d'origine.
- **La prochaine fois** : pour tout id Vectorize dérivé d'un chemin/texte variable, utiliser un hash court plutôt qu'un slug direct — un chemin de fichier peut dépasser la limite sans avertissement à l'écriture du code, seulement à l'exécution réelle contre l'API.

## 🎯 Le seul moyen de donner à des Cloudflare Functions un "accès fichier" statique est un snapshot généré et COMMITTÉ (pas de lecture filesystem en prod) — 2026-07-04
- **Contexte** : pour ingérer `docs/*.md` dans le RAG, impossible de faire un `fs.readFileSync` au runtime (les Pages Functions n'ont aucun accès filesystem en production, contrairement à un script Node local). Solution (déjà précédentée par `_featureDigest.js`) : un script Node dev-time (`scripts/generate-docs-digest.mjs`) lit les fichiers et écrit un module JS statique (`_docsDigest.js`, données en `JSON.stringify` dans une constante exportée), importé normalement par le bundler Functions comme n'importe quel autre module.
- **La prochaine fois** : pour toute nouvelle donnée "de fichier" qu'un agent/endpoint doit connaître en prod, ne PAS chercher un moyen de lire le filesystem au runtime — générer un module statique committé via un script dev-time, avec un rappel de régénération dans `/cloture-session` (le digest devient périmé si le contenu source change et qu'on oublie de relancer le générateur, mais ne casse jamais le build puisque committé).

## 🔴 Un LLM en sortie JSON structurée peut inclure un item tout en écrivant dans son propre champ qu'il ne devrait PAS être inclus — 2026-07-04
- **Piège trouvé en dry-run** (avant mise en prod) : `agents-triage.js` demandait au LLM un tableau JSON `[{id, reason}]` des items à bloquer. Résultat observé : certains items apparaissaient bien dans le tableau, mais avec un `reason` du type "pas de problème ici" / "mais pas exactement, donc pas de blocage" — le modèle exprime le doute dans le TEXTE tout en laissant l'item dans la liste "à bloquer" structurellement. Un parsing naïf (`if (item.id && item.reason) → bloquer`) aurait donc appliqué des blocages que le LLM lui-même jugeait incorrects.
- **La prochaine fois** : pour toute extraction JSON structurée d'un LLM qui classe/filtre des éléments, ajouter un filtre défensif POST-parsing qui rejette les entrées dont le champ texte contient des marqueurs de hedging/contradiction ("pas de problème", "pas de blocage", "correct ici", "légitime ici", etc.) — ne jamais faire une confiance aveugle à la présence structurelle dans le tableau. Combiner avec un garde-fou de proportion (si le LLM classe >X% du lot dans une catégorie à fort impact, c'est probablement un dérapage → tout annuler plutôt qu'appliquer).

## 🎯 Le plan AI-Ops (D1) prime TOUJOURS sur `MODELS` statique de `_llm.js` — 2026-07-01
`callLLM()` résout le modèle via `opts.model || plan.models?.[providerId]?.[tier] || MODELS[providerId]?.[tier]` — le plan D1 (construit par `/api/ai-ops` via discovery live + `RANK` de `ai-ops.js`) **écrase toujours** le fallback statique. Corriger `MODELS.groq` dans `_llm.js` après une dépréciation provider est nécessaire (filet de sécurité si le plan est absent/périmé) mais **insuffisant seul** — si le plan D1 existant référence encore l'ancien modèle, il continue à être utilisé jusqu'au prochain refresh (auto si `age_h > 20`, ou manuel `POST /api/ai-ops?secret=...&action=refresh`). **Réflexe après toute dépréciation Groq/provider : (1) corriger `RANK` dans `ai-ops.js` (retire l'entrée dépréciée) ET `MODELS` dans `_llm.js`, (2) forcer un refresh du plan, (3) vérifier `GET /api/ai-ops?secret=...` que le plan ne référence plus l'ancien modèle.** Vécu 2026-07-01 : `medium` pointait encore vers `llama-3.3-70b-versatile` dans le plan D1 malgré la correction statique, jusqu'au refresh manuel.

## 2026-06-21 — Reels IG+FB, pipeline auto-pub

- **IG Reels timeout ≠ échec définitif** : Meta prend jusqu'à 60s pour encoder une vidéo côté serveur ; CF Pages timeout = 30s → le container est créé MAIS le polling expire avant `FINISHED`. Solution : stocker le `container_id` dès l'étape 1 dans le résultat D1 ; au retry, appeler `publish_container` directement (container déjà encodé entre temps). Ne JAMAIS recréer le container à chaque tentative.
- **FB Reels = endpoint séparé** : `POST /{page-id}/video_reels` (pas `/{pageId}/feed`, pas `/{igId}/media`). Corps : `{ video_url, description, title, published }`. Distinct des posts photo/texte FB. `META_PAGE_ID` requis (déjà dans les secrets).
- **Dual-gate serveur-à-serveur** (confirmé) : `social.js POST` accepte Bearer admin OU `?secret=POSTSTAY_SECRET`. Quand `agent-drafts.js` appelle `/api/social` en interne, passer `?secret=` — le Bearer ne se propage pas entre Functions CF.
- **Fenêtre auto-publish : mettre ≥30j, idéalement 90j** — une fenêtre de 14j laisse silencieusement orphelins tous les drafts `approved` planifiés > 14j en arrière (typique après backlog ou pause). Symptôme : cron tourne mais ne publie rien. Diagnostic : simuler la query de sélection D1 manuellement.
- **`callLLM` retourne `{ok, text}` jamais une string** — ne JAMAIS écrire `captionRes.trim()` ou `captionRes.split()` directement sur le résultat. Toujours `captionRes.text.trim()` après avoir vérifié `captionRes.ok`.

- **`rebuildRevenus2026_` = DESTRUCTIF si le Sheet contient des données mixtes (système + manuel).** La fonction zéro TOUTES les lignes data des colonnes cibles AVANT de re-appliquer depuis les onglets source. Toute donnée saisie manuellement hors des onglets source (Airbnb historique, corrections, données hors-système) est définitivement perdue. **La prochaine fois : NE JAMAIS faire de zéro global sans avoir exporté une copie du Sheet.** L'alternative safe = trigger `syncRevenus2026` (mémo-based = incremental, jamais destructif).

- **Le trigger `syncRevenus2026` (15 min) est la seule voie safe pour les incréments.** Il utilise le mémo `rev2026_traites` pour ne traiter que les résas nouvelles, jamais les existantes. Il ne touche jamais les cellules qui n'ont pas de delta à appliquer. Utiliser TOUJOURS cette voie pour les nouveaux enregistrements ; la fonction rebuild est réservée à un reset total sur données 100% système.

- **Avant tout rebuild rétroactif sur un Sheet Google, demander : "ces données viennent-elles toutes d'un onglet source ?"** Si la réponse est "non" (données manuelles, corrections, imports externes), un rebuild qui zéro+reapply va détruire les données orphelines sans avertissement.

## 📣 Auto-publication réseaux — pièges du fact-check & du pipeline — 2026-06-15 (soir)
- **Le `META_PAGE_TOKEN` publie bien FB+IG** (`pages_manage_posts` / `instagram_content_publish` présents) MALGRÉ la régénération du matin pour le bot social — vérifié par un VRAI test de publication (post Bellevue id FB `986487064137992`). `debug_token` sur un **page token** renvoie `scopes:[]` (non concluant) → le seul test fiable des droits de publication = publier réellement. Ne jamais conclure « token ne peut pas publier » sans test live.
- **Le hashtag `#AmaryllisLocations` est un faux-positif systématique** pour toute règle fact-check `équipement.*amaryllis` (cascade/piscine + le mot « amaryllis » du hashtag de marque). Fix : **stripper les hashtags** (`/#[^\s#]+/g`) AVANT d'appliquer les règles factuelles — les hashtags sont du marketing, pas des affirmations.
- **Fact-check générique = aveugle au bien → faux positifs.** « piscine à débordement » est VRAI pour Amaryllis mais la règle flaguait tout. Solution : passer le `bienId` à `factCheckCaption(caption, rules, bienId)` + 2 mécaniques : `okFor:[biens]` (équipement légitime → skip) et `onlyFor:[biens]` (règle ciblée, ex « quatre suites » faux seulement pour amaryllis=3ch). Le `bienId` se dérive de l'imageUrl `/photos/{bien}/`.
- **Les mots bannis par bien (`agent_lessons.bien_id`) fuyaient en global** : `loadLearnedLessons` chargeait `pattern,reason` SANS `bien_id` → un « villa » banni pour Schœlcher bloquait TOUS les posts (dont Amaryllis qui EST une villa). Fix : charger `bien_id` → `rule.onlyFor=[bien_id]`. **Tout filtre/ban contextuel doit propager son scope bien jusqu'au point d'application**, sinon faux positifs massifs. Validé end-to-end : post Amaryllis frais score 88 → gate PASS → auto-approuvé en live (pipeline prouvé).
- **Les drafts « approved » ne repassent PAS le gate** (le cron `runEditorialAutoPublish` publie les approved tels quels). Donc tout post approuvé AVANT la mise en place du gate peut être faux. **Défense en profondeur indispensable** : remettre le fact-check BLOQUANT au point de passage unique = `executeDraft`/action publish dans `agent-drafts.js` (toute publication le traverse : cron, manuelle, gate).
- **Nomenclature villas (règle métier confirmée Vincent)** : SEULES Villa Amaryllis ET Villa Iguana sont des « villas ». Zandoli/Géko/Mabouya/Schœlcher(Bellevue)/Nogent ne le sont pas → règle `\bvillas?\b` onlyFor ces 5 biens.
- **CF Pages : un secret/var posé via `wrangler pages secret put` n'est actif qu'APRÈS un redéploiement** (vu avec `EDITORIAL_GATE_MODE` : resté shadow jusqu'au redeploy). Toujours redéployer après avoir posé une var Pages.
- **`handleSeed30Days` est idempotent** (anti-doublon sur dates déjà planifiées + exclut Iguana) → le re-seeder chaque jour à partir d'aujourd'hui maintient un horizon glissant 30j sans doublon = auto-reseed trivial.

## 🤖 Réseau d'agents : boucles d'autonomie — pièges rencontrés — 2026-06-15
- **`agent_memory` est injectée dans le prompt** (`buildPrompt` memorySection, agents-run.js) → écrire une clé dans `agent_memory(agent=<id>)` = le faire lire à l'agent au run suivant. C'est le canal de feedback le plus simple (utilisé pour `eval_feedback`). Le bus inter-agents = même mécanique avec `agent='_shared'`.
- **Sélection des sorties à évaluer : 1 par agent, pas les N plus récentes.** community-manager génère ~12 drafts/jour → il monopolisait les 25 places de `agents-eval` (les autres agents jamais notés). Fix : `JOIN (SELECT source, MAX(id) ... GROUP BY source)` → dernière sortie de CHAQUE source.
- **Batch d'éval : ordonner DESC + garde anti-écrasement.** En traitant plusieurs sorties du même agent dans un batch (ordre DESC), une vieille bonne note **supprimait** la correction d'une sortie récente faible. Fix : `Set` des agents déjà traités → seule la sortie la plus récente fait foi (vu en prod : feedback=1 puis effacé dans la même passe).
- **`no-empty` ESLint accepte un bloc avec commentaire.** Tout `catch {}` vide = +1 erreur lint → bloque le deploy (gate delta). Toujours mettre `catch { /* raison */ }`. Idem nouveau fichier : il part d'un baseline 0, donc 0 erreur tolérée.
- **Émettre un signal/donnée structurée d'un agent = champ JSON explicite, pas du parsing de prose.** On a ajouté un champ `signal` optionnel au schéma de sortie (parsé via `parsed.signal`), fiable et additif (absent = rien ne casse).

## Triage backlog — classes d'hallucination trouvées manuellement, non catchées par le cron — 2026-07-05

Le LLM avec une règle explicite + un exemple concret dans le prompt fait mieux qu'une heuristique regex fragile pour ce cas précis — contrairement à la détection de dette documentaire (endpoint manquant), qui elle EST un problème purement déterministe (cf. METHODOLOGIE-PROCESS.md).

## Le backlog `agent_actions` d'un agent fleet = source de vérité sous-exploitée (2026-07-08)
- **Le backlog D1 d'un agent (`agent_actions` filtré par `agent=<id>`) est souvent plus à jour que n'importe quel tracker manuel** — parce que l'agent lui-même y écrit `notes` à chaque run, y compris des correctifs de statut ("DÉJÀ FAIT — ..."). Avant de conclure qu'une feature liée à un agent fleet est "à construire", `GET /api/agents-actions?agent=<id>` (Bearer admin) et lire les `notes`, pas juste le `status` — un item peut afficher "bloqué" alors que sa propre note dit le contraire (statut jamais mis à jour après le vrai travail).
- **Un agent fleet peut IDENTIFIER un trou (le flaguer en backlog `bloqué`) sans jamais pouvoir le COMBLER lui-même.** Les agents personas (invoqués via l'outil Agent, ex. `veille-concurrentielle`) produisent de l'analyse/des recommandations, pas du code déployé — un item `agent_actions` de type "construire X" reste bloqué indéfiniment tant qu'une session de codage humaine (ou Claude en mode ingénierie) ne le prend pas en charge explicitement. Ne pas confondre "l'agent peut analyser ça" avec "l'agent peut le construire".

## Auto-fermeture du backlog IA (I-11, backlog-verify.js) — piège vécu au premier dry-run live (2026-07-18)
- **Une classification LLM sans mots-clés d'ancrage dans le texte source dérape vite vers un checkType plausible mais faux.** Premier test réel : le classifieur a mappé un item "Optimiser Core Web Vitals (LCP/CLS/INP)" sur `checkType=live_meta` (vérifie title/meta SEO) simplement parce que l'item citait une page — le lien entre la revendication (performance) et le check exécuté (SEO) était inexistant. Le prompt était pourtant explicite sur la définition de chaque `checkType`. **Une définition précise dans le prompt ne suffit PAS à empêcher un mapping plausible-mais-faux** — il faut une garde déterministe SÉPARÉE qui vérifie l'ancrage textuel (le nom d'event/schema/mot-clé apparaît littéralement dans l'item) avant d'accepter la classification. Généralise le principe "LLM propose, code déterministe dispose" à l'étape de CLASSIFICATION elle-même, pas seulement à l'étape de vérification finale.
- **Avant de construire un vérificateur "catégorie X = auto-checkable", inspecter la composition RÉELLE du backlog ouvert, pas seulement les items déjà fermés historiquement.** Hypothèse initiale (basée sur des items `fait` historiques comme des migrations de secrets) : les catégories seo/content/tracking/technique/performance seraient majoritairement des faits techniques vérifiables. Vérification live : 17/19 items *actuellement ouverts* dans ces catégories sont en fait de la production physique/créative (photos, Reels, témoignages) — la catégorie D1 ne prédit pas la nature de la revendication à l'intérieur. Sans ce contrôle, le système aurait ciblé un périmètre largement fictif.
- **Une automatisation d'observabilité ne doit JAMAIS s'écrire dans la métrique qu'elle est censée améliorer.** `backlog-verify.js` ferme des items du backlog mais n'insère volontairement PAS dans `action_outcomes` (contrairement au PATCH manuel `agents-actions.js`) — cette table alimente `delegation-stats.action_cochee`, qui mesure explicitement la charge de VINCENT. Si la fermeture autonome y était comptée, la métrique ne pourrait jamais montrer une baisse liée à la délégation (elle recompterait le travail de la machine comme du travail humain). Log de traçabilité séparé (`backlog_autoverify_log`) au lieu de réutiliser la table existante.

## Auto-publish éditorial : ne JAMAIS bloquer une file sur son plus ancien élément, ni ressusciter un fact-check factuel — 2026-07-22
- **`runEditorialAutoPublish` reprend TOUJOURS la plus ancienne entrée `approved`** (`sort ASC, slice 0,1`). Si celle-ci ne progresse jamais (échec en boucle), elle monopolise la file → aucun post ne sort. **Règle : toute file "plus ancien d'abord" a besoin d'un compteur de tentatives + un plafond qui `failed`/archive l'élément coincé**, sinon un seul draft défaillant gèle tout le pipeline (vécu : posts arrêtés 4 jours à cause d'un carrousel Zandoli).
- **Un fact-check FACTUEL bloquant (`blocked:true`, ex. "clapotis sur les hauteurs", "cascade" sur un bien sans cascade) n'est PAS corrigeable en réessayant le même contenu.** `runEditorialRetry` le ressuscitait `failed`→`approved` indéfiniment. **La prochaine fois : distinguer les échecs corrigeables (score/mots à réécrire via improved_blocks) des échecs FACTUELS (archiver directement, jamais relancer).** Fix appliqué : `runEditorialRetry` archive tout `blocked:true`.
- **Les agents éditoriaux (community-manager) inventent régulièrement des détails factuellement faux** par bien (clapotis, cascade, piscine à débordement sur le mauvais bien). Le fact-check les attrape (bien), mais ça produit des drafts jetés. Dette : renforcer le grounding par bien (`_biens.js` EQUIP_RULES) dans le prompt de génération pour réduire ces rejets à la source.
