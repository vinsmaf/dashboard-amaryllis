## Outillage / vérification (piège majeur, 2026-07-19)
- **Le preview browser intégré (`mcp__Claude_Browser__*`) ET `curl` n'exécutent PAS le JavaScript/React du site.** Sur une SPA, ils ne montrent que le HTML statique (le bloc `data-prerender-seo` dans `#root`), donc **toute route affiche "l'accueil"** et un screenshot peut être **blanc**. → **Ne JAMAIS conclure "cette page est cassée" sur la base du preview intégré ou de curl.** Signal fiable : `window.__BUILD__` (défini par `main.jsx`) est `null` dans le preview = le bundle n'a pas tourné. Le **vrai Chrome (`mcp__claude-in-chrome__*`)** est le seul juge d'une SPA — il a confirmé en 30 s que `/services/<bien>` marche parfaitement (perte de temps évitée : j'avais cru à un bug de routing prod inexistant, j'ai même déployé une page de contournement `depart-tardif.js` sur ce faux diagnostic).

## Meta Marketing API
- **Insights** : `time_increment=all_days` (pas `all` → 400 "must be monthly/all_days"). Fields perfs : `spend,impressions,clicks,actions,action_values` ; les conversions sont dans `actions[]` type `offsite_conversion.fb_pixel_purchase` (+ alias), la valeur dans `action_values[]`.
- **Carrousel** : `object_story_spec.link_data.child_attachments[]` avec `picture` (URL publique — **pas besoin d'image_hash**), `multi_share_optimized:false` pour garder l'ordre choisi. ⚠️ affichage **1:1 (crop carré)** → vérifier le cadrage des photos paysage/portrait.
- **Erreurs de création** rencontrées (toutes en PAUSED, cf. footgun) : `is_adset_budget_sharing_enabled:false` requis (ABO), `advantage_audience:0` (sinon impose âge ≤25/≥65 qui écrase le ciblage), `excluded_geo_locations.countries` = codes ISO en TEXTE (pas objet), "add a valid payment method" = pas un bug code (moyen de paiement compte). Toujours remonter l'objet `error` complet, pas `.message`.
- **Dédup/refresh** : le moteur crée une ad à chaque run → sans `findExistingAds` on duplique. `refreshCreatives:true` = graphDelete ad→créative (ordre : ad d'abord, une créative rattachée à une ad vivante ne se supprime pas) puis recrée.
- **Meta a 3 niveaux à activer, indépendants : campagne → ad set → ANNONCE.** Activer la campagne + les ad sets dans Ads Manager NE suffit PAS : si l'annonce (ad) reste PAUSED, **rien ne diffuse** (vécu 2026-07-19, Vincent croyait avoir "lancé" alors que les 2 ads étaient PAUSED). `effective_status` de la campagne peut être ACTIVE alors que les ads sont PAUSED. Toujours vérifier les 3 niveaux (`?debug=status` lit campagne+adsets+ads). Nos objets sont créés PAUSED → l'utilisateur doit activer explicitement l'onglet "Publicités" aussi.
- **Un ad set à créative cassée (post FB supprimé, `error_subcode 2446289` "Ad Creative Is Incomplete") n'est PAS supprimable par l'API Meta**, même après avoir purgé ses annonces enfants (vécu 2026-07-19 sur 3 ad sets parasites). Contournement API = échec net. **Solution : supprimer l'objet dans l'UI Ads Manager** (elle gère les objets cassés) OU supprimer la campagne entière (emporte tout). Quand une campagne a des objets cassés qu'on veut nettoyer + peu/pas d'historique d'apprentissage (spend~0) → **le plus simple et propre = supprimer toute la campagne dans l'UI et la recréer de zéro depuis le brief** (source unique déterministe, zéro parasite).
- **Idempotence par nom = fragile si l'utilisateur renomme la campagne dans Ads Manager.** `findExistingCampaign` matche sur le nom exact ; Vincent a renommé "C1 — TOFU Découverte" → "Amaryllis campagnes" et TOUS les endpoints (status/prune/sync/create) ont cassé ("introuvable"). Fix = aligner `CAMPAIGNS[key].name` du brief sur le libellé réel. Outils de diagnostic ajoutés : `?debug=status` (effective_status campagne+adsets+ads), `?debug=list` (toutes les campagnes du compte), `POST action=prune` (supprime les ad sets hors-brief, garde-fou nom exact, dry-run par défaut).
- **Race de propagation edge Cloudflare après déploiement** : deux appels consécutifs au même endpoint peuvent taper des bundles différents (un nœud sur l'ancien code, un sur le nouveau) pendant ~1-2 min post-déploiement. Vécu : un `POST prune confirm` a échoué avec l'ANCIEN nom de campagne juste après un dry-run réussi avec le nouveau. Le garde-fou (match nom exact) fait échouer PROPREMENT plutôt que d'agir à tort → retry en boucle jusqu'à ce que le bon bundle réponde.

## Google Ads API — setup (2026-07-19)
- **Developer token** ≠ tracking conversion. Ce sont 2 choses distinctes souvent confondues : le tracking (import GA4 `purchase` → conversion Google Ads) était **déjà en place** ; le developer token (accès API par script) n'avait jamais été demandé.
- **Le developer token se demande UNIQUEMENT depuis un compte administrateur (MCC)**, jamais un compte Ads simple (message "Seuls les comptes administrateur ont accès au centre API"). Chemin complet : créer un MCC (gratuit) → y lier le compte Ads (demande depuis MCC + acceptation côté compte) → API Center du MCC → accepter les CGU API → token en statut **"Accès Explorer"** (test only) → **demander l'accès "Basic"** (formulaire `support.google.com/adspolicy/contact/new_token_application` : exige un **numéro de projet Google Cloud** + un **document de design PDF** + review Google ~5 j).
- **Règle tenue** : sur Google Ads, Claude reste en **lecture seule** (navigation/lecture via claude-in-chrome), **jamais de clic d'action** (lier compte, accepter CGU, soumettre) — c'est Vincent qui exécute, Claude guide + rédige le contenu (texte des champs, PDF de design).

## "0 conversion" sur un canal payant — diagnostic (2026-07-19)
- **Ne jamais conclure "tracking cassé" ni "canal à couper" sur le seul rapport Google Ads.** Croiser avec **GA4 direct** (`/api/analytics` → `byChannel`, `funnel`) : GA4 dit honnêtement combien d'achats par canal, indépendamment de l'import Ads. Vécu : le "0 conversion Google Ads" (509€/30j) semblait un tracking cassé ; en fait la **config était saine des 2 côtés depuis juin** (GA4↔Ads lié 30/05, conversion `purchase` importée + "Action principale" 02/06). Le vrai diagnostic = **échantillon minuscule** : 6 achats sur TOUT le site en 30j, taux ~0,6% → attendu théorique ~1,5 achat pour 252 sessions Paid Search → 0 au lieu de 1-2 = **variance normale, pas un bug**. → attendre plus de volume avant de trancher un budget.
- **`purchase` (et autres events e-commerce standards) sont verrouillés "événement clé" dans GA4** — non démarquables ("Impossible de ne plus marquer l'événement clé"). Leur étoile s'affiche grisée ≠ inactif. Ne pas confondre avec un event custom démarquable.
- **La case "Unassigned" de GA4 `byChannel` peut concentrer un gros CA** (vécu : 4 achats / 4 244€, 2ᵉ poste de revenu) = sessions dont l'attribution s'est cassée en route (souvent la redirection de paiement Stripe qui perd le canal d'origine). Signal d'attribution à surveiller, pas forcément du Google.

## Brique 3 — exécution plafonnée (garde-fous encodés dans le code)
- **Un garde-fou "jamais faire X" doit être STRUCTUREL, pas juste documenté.** Pour l'agent budget pub : au lieu d'écrire "ne jamais activer un ad set", on s'assure qu'**aucune fonction ne SAIT le faire** (les seules primitives d'écriture Meta exposées = `pauseAdSet` + `setAdSetBudgetCents` ; il n'existe pas de `resumeAdSet`/`createCampaign` appelable). Résultat : même un bug/verdict aberrant ne peut pas déclencher une dépense nouvelle. `planExecutionAction` n'a aucune branche retournant "activate".
- **Plafond global dur ≠ plafonds par-bien.** Répartir 600€ par bien ne garantit PAS que la somme ≤ 600€ si un bien a plusieurs ad sets. Vincent a exigé un plafond sur la SOMME → `enforceGlobalMonthlyCap` rabote les hausses (jamais les baisses) pour que la somme des budgets mensuels des ad sets actifs reste ≤ budgetMax, testé sur plusieurs hausses simultanées.
- **"Gérer à la baisse comme à la hausse" ≠ pause sèche.** Baisse graduelle -25%/jour (`decrease_budget`) qui converge vers la pause sous un plancher (3€/j) : un ad set durablement non rentable décroît en quelques jours au lieu d'être coupé d'un coup ; un rentable monte +20%/jour capé (bien + global). Auto-convergent, tourne en cron quotidien.

## Modèle CAC (budget pub) — RM-08
- **CAC plafond PAR BIEN = 50% de la commission Booking nette évitée** = `prix × nuits(≈4) × (0.16 − 0.015) × 0.5`. Amaryllis ~81€, Zandoli/Géko ~32€, Mabouya ~20€. Seuil viable ~30€ (coût réel d'acquisition d'une RÉSA, pas d'un clic) → **paid seulement sur Amaryllis/Zandoli/Géko**, zéro sur Mabouya/Bellevue/Nogent. Contre **Booking** l'arbitrage est large (16%), contre **Airbnb** il s'effondre (host-fee ~3% côté hôte).
- **⚠️ Erreur du facteur 6 dans les vieux docs** (`PLAN-EXECUTION`, `METRICS_H1`) : "+1 point de direct = +1 600€/an" compte le **CA déplacé** comme s'il était de la marge. FAUX : basculer 1 point OTA→direct ne gagne que la **commission évitée** (~250€/point), pas le CA. Le seul doc cohérent était la reco Nogent (distingue CA direct et commission économisée). Ne jamais calibrer un budget pub sur le CA déplacé.

## Cloudflare Functions (robustesse)
- Un `fetch` externe (Graph API) qui throw dans une Function **sans try/catch = 502 edge muet** (text/plain "error code: 502", illisible). Toujours : try/catch global + `AbortController` timeout + parser `r.text()` puis `JSON.parse` en try (une réponse non-JSON ne doit pas crasher). Remonter l'erreur en **JSON status 200** (`{ok:false,error}`) pour la lire.
