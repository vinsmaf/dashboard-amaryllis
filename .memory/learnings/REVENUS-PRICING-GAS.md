## ✅ Le "forwarding POST classique" de sheets-proxy.js fonctionne — le bug redirect Apps Script ne touche que les payloads TABLEAU — 2026-07-16
- **Contexte** : en préparant un fix pour `cancelReservations_`, hypothèse initiale que TOUT POST vers Apps Script perdait son body (bug RESA-001, déjà documenté pour `importAllReservations`) — donc que même `revenus2026RebuildBienApply`/`revenus2027RebuildBienApply` (déjà en prod côté `beds24-webhook.js`, via la branche générique de `sheets-proxy.js`) pouvaient être silencieusement cassés depuis toujours.
- **Vérifié avant de coder** : test live non destructif (`?action=revenus2026RebuildBienDry`, `apply:false`) via la branche générique — réponse exacte `{ok:true, mode:"dry", ...}`, PAS un payload `readAll_()` (ce qu'on aurait vu si le body était réellement perdu et que `doGet` tombait sur son action par défaut). Le POST générique marche bien pour les payloads SIMPLES (objet plat) ; seul un payload avec un TABLEAU (`reservations`, `annulations`) déclenche le bug.
- **La prochaine fois** : ne jamais supposer qu'un bug de transport connu s'applique à TOUS les payloads d'un même endpoint — tester empiriquement avec un appel sûr (dry-run si disponible, ou un id/uid inexistant) avant d'élargir le diagnostic. Aurait évité de reconstruire un mécanisme GET pour des actions qui marchaient déjà.

## 🔴 GAS : une fonction de recherche appelée en boucle par bien re-fetch les mêmes colonnes à chaque fois — cacher UNE FOIS par sheet, pas par appel — 2026-07-14/15
- **Contexte** : `findChargesTotalRow_`/`findHistTotalRow_`/`findHistNightsRow_` faisaient chacune `sheet.getRange(1,1,lastRow,1).getValues()` (colonne A entière) + colonne B, À CHAQUE appel — appelées 9× par bien (parfois ×4 années via `readHist_`) dans une seule requête `read()`. Total : 100+ lectures réseau redondantes vers Google Sheets pour une SEULE requête admin, alors que le contenu de ces colonnes ne change jamais entre les 9 appels d'une même exécution.
- **Signal qui aurait dû alerter plus tôt** : le dashboard affichait "⚠ Seed local" (badge rouge = `doSync()` en échec, tooltip "Délai dépassé") de façon persistante — facile de blâmer un problème réseau/navigateur plutôt que de lire la fonction GAS elle-même pour un anti-pattern de performance.
- **La prochaine fois** : toute fonction `find*Row_`/recherche par mot-clé appelée en boucle sur le MÊME sheet doit accepter un paramètre optionnel `cols`/`precomputed` — calculé UNE FOIS par le boucleur avant la boucle, jamais refetché à chaque itération. Symptôme à surveiller : une requête `read()`/agrégation qui devient progressivement plus lente avec le temps (le Sheet grossit → chaque re-fetch coûte plus cher) sans qu'aucune ligne de code n'ait changé.

## 🔴 `clasp deploy` peut renvoyer "Resource has been exhausted (e.g. check quota)" — signal FIABLE et définitif de quota Google épuisé, à tester avant de deviner — 2026-07-14
- **Contexte** : après plusieurs échecs de `/api/sheets-proxy` (timeout côté client, 502 côté écriture Beds24 le lendemain), plusieurs hypothèses tentantes (bug de code, session invalide, réseau) avant de tenter un simple `clasp deploy` — qui a lui-même échoué avec ce message Google explicite.
- **La prochaine fois** : si plusieurs endpoints Apps Script/Google échouent de façon inexpliquée le même jour après une session de tests intensive, tenter un `clasp deploy` (ou toute opération d'écriture Google légère et déjà fiable) AVANT de creuser plus loin dans le code — un message d'erreur de quota explicite de Google élimine instantanément toute hypothèse de bug applicatif et évite de chasser un fantôme.

## 🟡 Enrichir une résa via `enrichReservation_` (Sheet) ne persiste PAS côté D1 `direct_bookings` — un sync périodique D1→Sheet écrase le correctif en silence — 2026-07-15
- **Contexte** : une résa Booking.com enrichie manuellement (nom+prix posés via `enrichReservation` sur le Sheet) est réapparue avec son placeholder d'origine ("Voyageur Booking", 0€) après le déclenchement (volontaire, pour un autre bug) d'un re-sync `direct_bookings`→Sheet — parce que ce sync repousse TOUJOURS les valeurs de D1 en écrasement complet de la ligne (upsert par id), et D1 lui-même n'avait jamais été corrigé.
- **La prochaine fois** : pour toute résa dont l'`id` Sheet commence par `direct-` (= source D1 `direct_bookings`, qu'elle vienne de Stripe ou d'un import email Airbnb/Booking), une correction manuelle de nom/prix doit être écrite en D1 (`UPDATE direct_bookings ... WHERE payment_intent_id=...`) EN PLUS du Sheet, sinon elle sera écrasée au prochain sync (`fetchDirectBookingsAsEvents`/`trigger-sync.js` tournent régulièrement). Pour une résa purement iCal (`id` finissant par `@airbnb.com`/`@booking.com` sans préfixe `direct-`), le Sheet seul suffit — pas de source D1 amont qui écraserait.

## 🔴 3 notions/couches de « prix » à ne jamais confondre — confondues plusieurs fois dans la même session [2026-07-10, précisée 2026-07-18]
- **Le mot « prix » recouvre 3 sources différentes** : prix DE BASE (`src/data/biens.js`, 1 chiffre statique/bien, fallback uniquement), reco RM (`/api/rm-recommendations`, advisory only, PAS le prix affiché tant que Vincent ne l'a pas appliqué), et prix RÉEL affiché (`/api/site-config?type=prices` pour les 6 biens Martinique, `/api/beds24-rates` pour Nogent — édité depuis l'onglet Tarifs admin). Le calcul exact du site public est `dailyPricesMap[date] ?? bien.prix`.
- **Découvert le 2026-07-10 en construisant `parity-check.js`** (contrôle de parité tarifaire direct vs Booking) : j'ai proposé successivement le prix de base puis la reco RM avant que Vincent tranche explicitement pour le prix réel de l'onglet Tarifs — la confusion s'était déjà produite plusieurs fois avant dans la même session sur ce même sujet.
- **Précisé le 2026-07-18** : la chaîne de résolution réelle (`PublicSite.jsx` `loadDailyPrices()`) a en fait **3 couches**, pas juste 2 notions à choisir : override serveur (`/api/site-config?type=prices`) `??` seed calibré (`SEED_DAILY_PRICES`, `src/seedPrices.js`) `??` flat `bien.prix`. Erreur commise ce jour-là : conclure qu'Amaryllis n'avait "jamais de prix pour l'été 2026" en ne vérifiant QUE l'override serveur (couche 1), sans consulter le seed (couche 2) — un écart de reco RM annoncé à +121% qui était en réalité +26-44% une fois le bon prix trouvé. **Ne pas s'arrêter à la 1ère couche vide et conclure "pas de prix" — toujours suivre la chaîne complète jusqu'au bout.**
- **La prochaine fois** : toute question « quel est le prix du bien X à la date Y » = suivre la chaîne complète override ?? seed ?? flat ci-dessus (jamais `rm-recommendations` seul, jamais `biens.js` seul, jamais s'arrêter à la 1ère couche vide). Détail complet : `CLAUDE.md` section « 1bis. Pricing — 3 notions à ne JAMAIS confondre ».
- **Un moteur indépendant peut driver sans qu'aucun check ne le voie** (trouvé le 07-18) : le RM a sa PROPRE base saisonnière (`rm_seasonal_profiles.base_price_override`, D1), sans lien avec `SEED_DAILY_PRICES`. Audité : dérive de +50% (Jan-Avr) à -45% (Sep-Déc) selon le bien, invisible faute de comparaison — corrigé (60 lignes) + garde-fou permanent mensuel `/api/rm-seed-drift` (cf. ADR-RM-SEED-DRIFT-001). Règle générale : dès qu'un système a SA PROPRE copie d'une donnée censée refléter une source de vérité, prévoir un check de dérive récurrent dès la conception.
- **Récidive quasi immédiate, même jour (07-18)** : Claude a recommandé "Schœlcher sous-évalué, tester +15%" en citant l'ADR YTD (87€, moyenne bloquée sur `/api/revenue-summary`) — un chiffre qui mélange mois creux et mois pleins. Une vérif de la couche seed AVANT d'écrire l'override a montré que le prix EFFECTIF proche (juil-août) était déjà 128-154€, largement au-dessus de 87€×1.15. Chantier abandonné à temps, aucun override écrit. Confirmé par Vincent : la baisse observée fin août-septembre (override 20-25€ sous le seed) est volontaire et correcte ("septembre très peu de demandes"). Règle : un ADR/occupation YTD est un indicateur de TENDANCE, jamais une base de calcul pour une action de prix.

## 🔴 `rebuildRevenus2026_` ne recalcule JAMAIS un mois déjà passé — un total affiché peut diverger du détail sans qu'aucune erreur ne le signale — 2026-07-09
- **Découvert en vérifiant un écart de 8,9% sur Amaryllis** (revenus locatif 2026 vs somme des résas) : `rebuildRevenus2026_` (`REVENUS_AUTO_2026.gs`) a `fromMonth` par défaut = mois COURANT (`currentMonthDefault_()`), et le commentaire du code dit explicitement "PAS de rebuild complet : Toutes les Reservations ne remonte qu'à avril 2026 (jan-mars ont péri de la source)" — donc tout mois antérieur au mois courant est une valeur FIGÉE depuis son dernier rebuild manuel, jamais recalculée automatiquement même si le détail source change ensuite (résa corrigée après coup via `patch-booking.js`, résa ajoutée en retard...). Confirmé concrètement : une résa Booking (Olivier Pointeau, 5 073,84€, réservée dès avril 2025) était comptée dans le total janvier depuis toujours, mais totalement absente du détail "Toutes les Réservations" — sans elle, l'écart semblait être un vrai bug ; avec elle réinjectée manuellement, l'écart de janvier est retombé à ~2%.
- **La prochaine fois** : ne jamais comparer "revenus locatif 2026" (agrégat) et "Toutes les Réservations" (détail) pour un mois PASSÉ en supposant qu'ils devraient être identiques — un écart peut simplement signifier "une correction a eu lieu après le dernier rebuild de ce mois", pas une erreur de calcul. Un rebuild `fromMonth=1` corrigerait la divergence MAIS risque de faire BAISSER les totaux si des résas ont vraiment disparu de la source depuis (jan-mars) — ne jamais le lancer sans validation explicite de Vincent, dry-run d'abord (`revenus2026RebuildBienDry`, sans risque, montre juste la portée pas les nouvelles valeurs).
- **Format des montants OTA** : le Sheet stocke le montant NET (après commission plateforme), pas le brut visible par le voyageur — vérifié via les relevés officiels Airbnb (mensuel, agrégé par logement, donne directement le net) et les PDF "Détails de la réservation" Booking.com (`montant soumis à commission` − `commission et frais` = net). Toute résa OTA saisie manuellement doit utiliser CE net, jamais le prix affiché au client.

## 🔴 Une ligne de sous-total codée en dur (`revRow` fixe) ne survit pas à une structure de Sheet qui diffère d'un onglet à l'autre — 2026-07-07
- **Piège trouvé en contestant un "0€" que Vincent savait faux** : `readHist_()` (`SCRIPT_SHEETS.js`) lisait la ligne TOTAL de chaque bien via un numéro de ligne FIXE (`BIENS_MAP.revRow`), supposé identique sur tous les onglets années (2022→2025). Mais les onglets historiques ont une ligne "Parking" en plus sous Nogent que l'onglet 2026 (courant) n'a pas — décalant de +2 la ligne TOTAL de CHAQUE bien suivant dans l'ordre du Sheet, sur 4 onglets entiers, silencieusement (aucune erreur, juste une valeur d'une autre ligne — parfois 0, parfois un nombre plausible mais faux).
- **La prochaine fois** : pour toute lecture de "ligne X du bien Y" dans un Sheet géré manuellement par un humain (structure qui peut driver d'un onglet à l'autre au fil des ans), ne jamais coder le numéro de ligne en dur — chercher dynamiquement (mot-clé colonne A + 1ère cellule "TOTAL" dans les N lignes suivantes en colonne B). Un numéro de ligne fixe qui a raison sur l'onglet le plus récent (celui qu'on teste en premier, par réflexe) peut être silencieusement faux sur tous les autres.
- **Leçon méthodo associée** : après le fix, un signal résiduel ("Schœlcher toujours à 0€") a semblé indiquer un bug non résolu — c'était en fait ma propre erreur de lecture d'un screenshot antérieur (j'avais attribué la valeur de juin à juillet). Avant de creuser un "bug" qui persiste après un fix qui semblait complet, revérifier la lecture humaine des preuves déjà collectées plutôt que d'assumer que le fix est incomplet.

## 🔴 Toute Date lue depuis une cellule Google Sheet en Apps Script doit utiliser les composantes LOCALES, jamais `getUTC*` — 2026-07-04
- **Piège trouvé en écrivant un test anti-drift** : `nd_()` (fonction interne à `contentKeyRow_`/`contentKeyRow27_` dans `REVENUS_AUTO_2026/2027.gs`) lisait une Date de cellule Sheet via `getUTCFullYear()`/`getUTCMonth()`/`getUTCDate()`. Or `toNoonUTC_()` du MÊME fichier (et `SCRIPT_SHEETS.normDate_`) utilisent délibérément les composantes LOCALES (`getFullYear`/`getMonth`/`getDate`) — parce qu'Apps Script convertit une cellule formatée date en objet Date natif calé sur MINUIT heure Europe/Paris (fuseau du classeur), pas minuit UTC. Utiliser `getUTC*` sur cet objet recule silencieusement d'un jour (ex. cellule affichant "04/07/2026" → `getUTCDate()` renvoie 3 en heure d'été Europe/Paris).
- **La prochaine fois** : dans TOUT code Apps Script de ce projet qui traite une valeur susceptible d'être un objet `Date` issu de `sheet.getRange(...).getValues()` (pas une string), toujours utiliser les composantes locales (`getFullYear`/`getMonth`/`getDate`), jamais `getUTC*` — la seule exception documentée est `src/utils/resaDedup.js` côté runtime JS (Cloudflare Functions), qui lui reçoit des inputs déjà normalisés en string et n'a jamais affaire à une vraie cellule Sheet.

## 🔴 `addReservation_` (GAS) ne resynchronise JAMAIS les revenus sur une résa MODIFIÉE, seulement sur une résa NOUVELLE — 2026-07-03
- **Piège** : `addReservation_` a 3 branches de dédup — (1) id exact match → "updated" (ligne écrasée en place), (2) même contenu (bien|checkin|checkout) → "updated, merged:true" (ligne écrasée, ancien id préservé), (3) `sheet.appendRow(row)` → "added" (vraie nouvelle ligne). **`syncRevenus2026()` n'est appelé QUE dans la branche (3)`.** Résultat concret : en modifiant la résa d'Ary Augustin (Zandoli, 6→4 nuits, 898€→710€) via un id existant → branche "updated" → le Sheet "Revenus locatifs 2026" a gardé l'ANCIEN montant (898€) pendant toute la session, sans aucune erreur ni signal — jusqu'à ce que Vincent demande explicitement "et sur les revenus 2026 ça n'a pas créé de doublon ?".
- **La prochaine fois** : après TOUTE modification de prix/dates d'une résa existante (pas seulement création/annulation), relancer `revenus2026RebuildBienApply` (ou l'équivalent dry d'abord) pour le bien/mois concerné — ne jamais supposer que `syncRevenus2026()` a suivi automatiquement. Le vrai risque n'est pas le doublon (dédup id/contenu solide) mais le **chiffre périmé silencieux**.

## 🟡 Un défaut temporel codé en dur dans un script "safe/idempotent" rote silencieusement — se donner un défaut auto-actualisé — 2026-07-03
- **Piège évité** : `rebuildRevenus2026_` avait `fromMonth` par défaut = `4` (avril), choisi car "avril = 1ère arrivée en source" **au moment où le code a été écrit**. Ce chiffre n'a plus jamais bougé. Si un appel avait omis `fromMonth` (oubli humain, script tiers, cron futur), le rebuild aurait re-zéroté et recompute avril→décembre en écrasant des mois déjà clos (juin par ex.) — sans qu'aucun signal ne prévienne, puisque la fonction est justement conçue pour "écraser puis recalculer".
- **La prochaine fois** : pour tout script de recalcul/rebuild dont le "point de départ" est un mois/une date, préférer un défaut **calculé dynamiquement** (`new Date().getMonth()+1` dans ce cas, via `currentMonthDefault_()`) plutôt qu'une constante — ça protège perpétuellement les périodes déjà closes sans exiger de maintenance mensuelle. Garder la possibilité de forcer une valeur explicite pour les corrections volontaires de mois passés (ne pas bloquer ce cas d'usage légitime).

## 📅 GAS : les cellules Date sont des objets JS, pas des strings — 2026-06-27
- **Piège** : `r[4]` (colonne checkin) dans GAS retourne un objet `Date` JS, pas une chaîne. `String(dateObject)` → "Sat Aug 15 2026 00:00:00 GMT+0000 (UTC)" → `parseInt` sur les 4 premiers chars = NaN.
- **La prochaine fois** : toujours tester `if (raw instanceof Date)` et utiliser `Utilities.formatDate(raw, "UTC", "yyyy-MM-dd")`. Sinon `String(raw || "")` pour les autres types. Affecte toutes les fonctions GAS qui lisent des colonnes de dates (`deleteReservation_`, `cancelReservations_`, etc.).

## ⚡ CF Pages Function : `context.waitUntil` pour GAS en background après webhook — 2026-06-27
- **Pattern** : webhook Beds24 doit répondre vite (timeout 10s). GAS `rebuildRevenus` prend >5s → répondre 200 immédiatement, puis pousser le rebuild en background avec `context.waitUntil(Promise.all(jobs))`.
- **La prochaine fois** : chaque fois qu'une Pages Function reçoit un webhook et doit déclencher une opération lente (GAS, sync longue), utiliser `context.waitUntil` avec `fetch(sheetsProxy, ...)` → le CF runtime maintient la Function vivante jusqu'à la fin du rebuild, sans bloquer la réponse.

## 📅 iCal Worker : stocker {uid, checkout}, pas juste uid — 2026-06-27
- **Piège vécu** : Géko a reçu une fausse alerte d'annulation (UID dont le préfixe était le même qu'une résa terminée). Root cause : KV stockait des UIDs sans date → impossible de savoir si la résa était terminée ou si Airbnb avait juste roté l'UID.
- **La prochaine fois** : `ICAL_STORE` key `uids:{bienId}:{canal}` = array de `{uid, checkout}`. Deux filtres à appliquer avant de marquer une annulation : (1) même préfixe (avant `-`) dans les résas actuelles = rotation UID ignorée ; (2) checkout < today = séjour terminé ignoré. Backward-compatible : old strings → `{uid, checkout: null}`.

## 🏷️ RM : passer `daily_floors` au `/calculate` = pattern correct pour respecter les prix réels — 2026-06-26

## 🏷️ RM D1 `rm_properties` : base_price_* = NULL par défaut — plancher = siteMinCents seulement — 2026-06-26

## 💰 Revenus Apps Script : MIN_AUTO_MONTH=6 — les arrivées avant juin jamais auto-traitées — 2026-06-26
- `syncRevenus2026_()` ignore les arrivées < juin même après `PurgeZero`. Pour corriger une résa avec arrivalMonth < 6 : soit `revenus2026ManualPatch_` (cellule unique), soit `rebuildRevenus2026_(apply, fromMonth, bienFilter)` (rebuild chirurgical par bien).

## 💰 Revenus Apps Script : le memo empêche le re-sync des résas supprimées — 2026-06-26
- Supprimer une résa du Sheet via `deleteReservation` NE retire PAS son ID du memo → le sync ne la retraitera jamais. C'est voulu pour les annulations (ex: Amaryllis Booking déc annulée → reste à 0€ même si l'iCal la repousse).
- Si on veut forcer le re-traitement d'une résa → purger son ID du memo (`revenus2026PurgeZero_` ou DELETE D1 direct).

## ⚠️ Apps Script revenus2026 : `apply:false` via sheets-proxy N'EST PAS un dry-run fiable — 2026-06-19
- **Piège vécu (donnée compta corrompue)** : appel `{"action":"revenus2026FromMonth","apply":"false","ignoreMemo":"true"}` via `/api/sheets-proxy` → la réponse a retourné `mode:"applied"` (PAS dry !) et a recompté 11 résas juillet→déc en **ignorant la mémoire anti-doublon** → **double comptage** (le système `appendCell_` ADDITIONNE, n'écrase pas). ## SQLite `INSERT...ON CONFLICT DO UPDATE` valide les NOT NULL AVANT le conflit — 2026-07-06
- **Passer `null` dans les VALUES d'un upsert échoue même si le `ON CONFLICT DO UPDATE` utilise `COALESCE(excluded.col, table.col)` pour "protéger" la colonne** — SQLite valide la contrainte NOT NULL sur les VALUES brutes de l'INSERT AVANT d'évaluer s'il y a conflit, donc l'upsert échoue silencieusement avant même d'atteindre le UPDATE. Root cause de la déconnexion OAuth Gmail/Calendar récurrente (`_googleOAuth.js` appelait `saveOAuthTokens(refreshToken: null)` à chaque refresh d'access_token).
- **Règle : tout upsert avec une colonne NOT NULL qu'on ne veut PAS écraser doit passer par un chemin `UPDATE` dédié** (sans jamais construire un INSERT avec `null` pour cette colonne), pas compter sur le `ON CONFLICT` pour "réparer" après coup.

## ⚠️ `ignoreMemo:true` (revenus 2026/2027) = risque de DOUBLE-COMPTAGE — 2026-07-19
- Dans `REVENUS_AUTO_2027.gs` (idem 2026), un ID présent dans le journal (`memo`) signifie **que la résa A DÉJÀ ÉTÉ ÉCRITE** dans l'onglet revenus. Ce n'est PAS un "ID mémorisé sans écriture".
